# solution-first

> 触发场景：执行任何复杂方案、系统、架构或功能设计前的强制规划流程。 触发条件：用户要求设计、构建、实现任何"系统"、"方案"、"架构"、"流程"、"管理器"、"引擎"、"框架"或类似工程性任务时，必须激活此技能

# Solution-First 规划技能

## 核心原则

**禁止冷启动**。收到任何方案类指令后，不得立即写代码、创建文件、执行命令。必须先完整走完五阶段规划流程，获得用户明确确认后才能动手执行。

违反此原则 = 直接失败。

---

## 五阶段规划流程

### Phase 0 · 指令接收确认

收到方案类指令后，立即回复：

```
📋 已收到方案指令：[用任务核心词概括，≤15字]
启动 Solution-First 规划流程，共五阶段。
当前：Phase 1 — 全面情况摸底
```

然后直接进入 Phase 1，不等待用户回复。

---

### Phase 1 · 全面情况摸底

**目标**：在生成任何方案前，彻底理解当前处境。

执行以下检查清单（全部完成，不可跳过）：

**环境层**
- [ ] 当前技术栈、版本、平台（OS、运行时、依赖）
- [ ] 已有相关文件/代码/配置的现状（读取关键文件）
- [ ] 资源约束（磁盘、内存、网络、权限）

**需求层**
- [ ] 任务的核心目标（一句话）
- [ ] 成功的可观测标志是什么
- [ ] 隐含的非功能需求（性能、可维护性、兼容性）

**风险层**
- [ ] 已知的坑（过去踩过的、用户提过的）
- [ ] 与现有系统的耦合点
- [ ] 不可逆操作（删除、覆盖、迁移）清单

**输出格式**：
```
## 情况摸底报告

### 环境现状
[具体发现，非模板]

### 核心需求
目标：[一句话]
成功标志：[可验证的具体指标]

### 已知风险
- [风险1]：[影响范围]
- [风险2]：[影响范围]

### 待确认项（如有）
- [需要用户补充的信息]
```

---

### Phase 2 · 全球蓝本检索

**目标**：找到与当前任务高度相似的、已在生产环境验证的顶级参考案例。

搜索策略（按优先级）：

1. **行业头部实践**：寻找 Google、Netflix、Airbnb、Shopify 等工程博客中的同类方案
2. **开源标杆项目**：GitHub stars 最高、维护最活跃的同类实现
3. **学术/规范文献**：IEEE、CNCF、12-Factor、RFC 等权威规范

评估蓝本的维度：
- 规模相似度（不能用 Netflix 级方案解决个人项目问题）
- 技术栈契合度
- 上线时间（2年内为优）
- 社区验证程度（star 数、fork 数、issue 活跃度）

**输出格式**：
```
## 蓝本选择报告

### 选定蓝本
名称：[项目/文章名]
来源：[URL 或出处]
规模匹配度：[高/中/低] — [原因]
技术契合度：[高/中/低] — [原因]

### 蓝本核心设计决策
1. [决策1]：[做了什么，为什么这么做]
2. [决策2]：...

### 与当前任务的差异点
- [差异1]：需要适配 → [适配策略]
- [差异2]：...

### 备选蓝本（如有）
[简要列出未选中的原因]
```

---

### Phase 3 · 验收标准设计

**目标**：在动一行代码之前，先定义"完成"的精确含义。

验收标准必须满足 SMART 原则：
- **Specific**：具体可描述
- **Measurable**：有数字或状态可测量
- **Achievable**：在当前约束内可达
- **Relevant**：与核心目标直接相关
- **Time-bound**：明确完成时间预期

**验收标准分三层**：

```
## 验收标准

### L1 · 最低可用（Must Have）
必须100%满足，否则方案失败：
- [ ] AC-01：[具体可测试的条件]
- [ ] AC-02：...

### L2 · 完整达标（Should Have）
正常情况下应满足：
- [ ] AC-03：...
- [ ] AC-04：...

### L3 · 卓越指标（Nice to Have）
超出预期的加分项：
- [ ] AC-05：...

### 验收方法
每条AC对应的测试/验证命令或检查步骤：
- AC-01 验证方式：[具体命令或操作]
- AC-02 验证方式：...
```

---

### Phase 4 · 落地执行方案

**目标**：生成完整的、可直接执行的方案文档，并保存到指定目录。

**方案文档结构**：

```markdown
# [任务名称] 落地执行方案
生成时间：[timestamp]
蓝本参考：[Phase 2 中选定的蓝本]

## 执行概览
- 预估工时：[X小时]
- 风险等级：[低/中/高]
- 不可逆操作：[列表或"无"]

## 架构设计
[基于蓝本适配后的架构图或结构说明]

## 分步执行计划
### Step 1：[步骤名]
- 操作：[具体命令/代码/动作]
- 预期结果：[可观测的成功标志]
- 失败处理：[如果出错怎么办]

### Step 2：...

## 关键代码/配置
[核心实现的骨架，非完整代码]

## 风险预案
| 风险 | 触发条件 | 应对措施 |
|------|----------|----------|
| ... | ... | ... |

## 验收执行步骤
[按 Phase 3 验收标准逐条给出验证命令]
```

**保存方案文档**：

```python
# 路径规则
plan_dir = r"E:\AI\Workspace\AI方案"
filename = f"{task_name}_{timestamp}.md"
full_path = os.path.join(plan_dir, filename)
```

保存后输出：
```
✅ 方案文档已保存：E:\AI\Workspace\AI方案\[文件名]
```

### Phase 5 · 等待确认，按方案执行

方案生成完毕后，输出确认提示：

```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
📄 方案文档：E:\AI\Workspace\AI方案\[文件名]
📊 共 [N] 个执行步骤 | [N] 条验收标准 | 预估 [X] 小时

请确认后我开始执行。可以：
• 直接回复"确认执行" → 按方案全速推进
• 回复"修改[具体内容]" → 我更新方案后再确认
• 回复"只做Step N-M" → 只执行指定步骤
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

**收到确认后**：
- 逐步执行，每完成一个 Step 输出进度
- 遇到偏差立即暂停，报告情况并询问
- 全部完成后，逐条运行验收检查，输出通过/失败状态

## 特殊情况处理

### 紧急小改动
若用户明确说"就改一行"、"快速修个bug"、"只是加个参数"等明确的小范围修改，可跳过 Phase 1-2，但仍需：
1. 说明要改什么
2. 说明潜在影响
3. 获得确认

### 用户催促跳过
若用户说"不用写方案直接做"、"跳过规划"：
回复：
```
明白您想快速推进。不过跳过规划在复杂任务中通常会造成更多返工。
建议：我用5分钟完成精简版规划（Phase 1+3），然后立即执行？
还是您确认要直接开始？
```

若用户坚持，按其意愿执行，但在执行日志开头标注 `[⚠️ 跳过规划流程]`。

## 方案目录管理

默认方案存储路径：`E:\AI\Workspace\AI方案\`

文件命名规则：`[任务简称]_[YYYYMMDD_HHMM].md`

示例：`技能生命周期管理系统_20250614_1430.md`

如目录不存在，自动创建：
```bash
mkdir -p "E:\AI\Workspace\AI方案"
# Windows:
New-Item -ItemType Directory -Force -Path "E:\AI\Workspace\AI方案"
```
